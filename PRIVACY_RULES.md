
# 🔐 Правила приватности и защиты данных

Этот документ описывает, КАК мы защищаем данные пользователей. Главная задача: приватность by design — защита встроена в архитектуру, а не прикручена потом.

---

## 🎯 Почему это важно

Финансовые данные — это самое личное, что есть у человека:
- Сколько он зарабатывает
- Куда тратит деньги
- Какие у него долги
- Где хранит сбережения

Если мы не защитим эти данные — мы не заслуживаем доверия. Одна утечка — и репутация потеряна навсегда.

---

## 📋 Основные правила

### 1. `user_hash` вместо PII (персональных данных)

**Что такое PII**:
- Email
- Телефон
- ФИО
- Паспортные данные
- Номера банковских карт
- Адреса

**Правило**: В логах, аналитике и AI-промптах используем ТОЛЬКО `user_hash` — анонимный идентификатор (SHA-256 от `user_id` + `salt`).

**Пример**:
```
❌ НЕПРАВИЛЬНО:
logger.info(f"User {user.email} created transaction {amount}")

✅ ПРАВИЛЬНО:
logger.info(f"Transaction created", extra={"user_hash": user_hash, "amount": amount})


**Почему это важно**: Если логи попадут к злоумышленникам, они не смогут связать `user_hash` с реальным человеком.

---

### 2. Шифрование токенов банков

Когда пользователь подключает банк (Тинькофф, Сбер), мы получаем OAuth-токен для доступа к его данным.

**Правило**: Токены хранятся ТОЛЬКО в зашифрованном виде (AES-256-GCM или Fernet).

**Ключ шифрования**:
- НЕ хардкодить в коде
- Хранить в переменной окружения или секретном хранилище (Vault)
- Ротация ключей — по расписанию

**Пример**:
```
❌ НЕПРАВИЛЬНО:
db.save(oauth_token)  # Plaintext!

✅ ПРАВИЛЬНО:
encrypted_token = encrypt(oauth_token, encryption_key)
db.save(encrypted_token)


**Почему это важно**: Если БД будет скомпрометирована, токены останутся зашифрованными.

---

### 3. Локализация данных (серверы в РФ)

**Правило**: Все данные пользователей хранятся и обрабатываются на серверах в России.

**Почему это важно**:
- Требование 152-ФЗ (российский закон о персональных данных)
- Защита от экстерриториальных санкций
- Доверие пользователей

**Что это значит**:
- БД (PostgreSQL, Redis) — в РФ
- Файлы и логи — в РФ
- AI-провайдеры — с серверами в РФ или без сохранения промптов

## 🤝 Consent Management (управление согласиями)

### Что такое согласие

Когда пользователь подключает банк, он даёт согласие на:
- Чтение баланса
- Чтение транзакций
- Синхронизацию данных

**Правило**: Каждое согласие хранится отдельно и может быть отозвано в любой момент.

### Как работает отзыв согласия

**Сценарий**: Пользователь нажимает "Отключить Тинькофф"

**Что происходит**:
1. Токен доступа к Тинькофф немедленно инвалидируется
2. Запись в `user_consents` помечается как `revoked_at = NOW()`
3. Данные перестают синхронизироваться
4. Уже загруженные данные остаются (пользователь может их удалить отдельно)
5. Запись в `audit_log`: "Отозван доступ к Тинькофф"

**Почему это важно**: Пользователь всегда контролирует, к каким данным у нас есть доступ.

## 📤 Экспорт данных

### Право на портативность

Пользователь имеет право получить все свои данные в машиночитаемом формате.

**Что экспортируем**:
- Все транзакции
- Цели и прогресс
- Категории и правила
- Настройки профиля

**Что НЕ экспортируем**:
- OAuth-токены банков
- Системные логи
- `user_hash` (это внутренний идентификатор)

**Форматы**:
- **JSON** — для машинной обработки (импорт в другие системы)
- **CSV** — для открытия в Excel
- **PDF** — для человека (красивый отчёт)

**Скорость**: Экспорт должен быть готов в течение 24 часов.

---

## 🗑️ Удаление данных

### Право на забвение

Пользователь имеет право удалить все свои данные.

**Как это работает**:

#### Шаг 1: Soft Delete (мягкое удаление)
- Пользователь нажимает "Удалить аккаунт"
- Все данные помечаются как `deleted_at = NOW()`
- Данные становятся недоступны через API
- OAuth-токены инвалидируются
- Запись в `audit_log`: "Запрос на удаление аккаунта"

#### Шаг 2: Период ожидания (30 дней)
- Данные физически ещё в БД (на случай ошибки/фрода)
- Пользователь может восстановить аккаунт (если передумал)
- Данные недоступны через API

#### Шаг 3: Hard Delete (полное удаление)
- Фоновая задача (cron) раз в сутки проверяет записи с `deleted_at < NOW() - 30 days`
- Физическое удаление из БД
- Запись в `audit_log`: "Данные полностью удалены"

**Почему 30 дней**: Защита от случайного удаления и фрода. Если пользователь передумает — можно восстановить.

## 📜 Аудит-лог (Immutable Ledger)

### Что такое аудит-лог

Это таблица, которая записывает ВСЕ важные действия пользователя:
- Вход в систему
- Подключение/отключение банка
- Экспорт данных
- Запрос на удаление

**Правило**: Аудит-лог — **immutable** (только INSERT, без UPDATE/DELETE).

**Что логируем**:
- `user_hash` — кто совершил действие
- `action` — что сделал (LOGIN, CONSENT_REVOKED, DATA_EXPORTED, ACCOUNT_DELETED)
- `resource_type` — к чему относится (provider, goal, transaction)
- `resource_id` — идентификатор ресурса
- `metadata` — дополнительная информация (без PII!)
- `created_at` — когда произошло

**Что НЕ логируем**:
- PII (email, телефон, ФИО)
- Токены
- Пароли

**Почему это важно**: Если возникнет спор ("я не удалял аккаунт"), у нас есть доказательство.

## 🎭 Маскирование PII

### В логах

**Правило**: Если в логе может появиться PII — маскируем его.

**Примеры**:
```
❌ НЕПРАВИЛЬНО:
logger.info(f"User {user.email} logged in from {ip_address}")

✅ ПРАВИЛЬНО:
logger.info(f"User logged in", extra={"user_hash": user_hash, "ip_hash": hash(ip_address)})
```

### В описаниях транзакций

Когда банк присылает описание транзакции, там может быть PII:
- "Перевод от ИВАН ИВАНОВИЧ И."
- "Оплата по телефону +79991234567"
- "Покупка в магазине МЕДИЦИНСКИЕ ТОВАРЫ"

**Правило**: Перед сохранением в БД маскируем PII:
- ФИО → `[PERSON]`
- Телефоны → `[PHONE]`
- Медицинские категории → `[MEDICAL]`
- Юридические категории → `[LEGAL]`

**Пример**:
```
Исходное описание: "Перевод от ИВАН ИВАНОВИЧ И."
После маскирования: "Перевод от [PERSON]"

**Почему это важно**: Если описание попадёт в AI-промпт или логи, PII не будет раскрыто.

## ⚠️ Edge-cases

| Ситуация | Обработка |
|----------|-----------|
| Пользователь просит "удалить меня прямо сейчас" | Объяснить про 30 дней, предложить soft delete |
| Пользователь хочет экспортировать данные другого пользователя | Отказать (только свои данные) |
| В описании транзакции PII | Маскировать перед сохранением |
| OAuth-токен истёк | Пометить как `revoked`, запросить новое согласие |
| Пользователь отзывает согласие, но хочет сохранить данные | Данные остаются, синхронизация прекращается |
| Аудит-лог заполнился | Архивировать старые записи (но не удалять) |

## ✅ Чек-лист готовности

При реализации защиты данных проверить:
- [ ] Все логи используют `user_hash`, не PII
- [ ] OAuth-токены зашифрованы (AES-256-GCM или Fernet)
- [ ] Ключ шифрования в переменной окружения, не в коде
- [ ] Серверы в РФ (Yandex Cloud, Selectel)
- [ ] Consent management реализован (таблица `user_consents`)
- [ ] Отзыв согласия → немедленная инвалидация токена
- [ ] Экспорт данных работает (JSON, CSV, PDF)
- [ ] Удаление данных работает (soft delete → hard delete)
- [ ] Аудит-лог реализован (immutable, только INSERT)
- [ ] Маскирование PII в логах и описаниях транзакций
- [ ] 152-ФЗ compliance (локализация, согласия, удаление)
```
