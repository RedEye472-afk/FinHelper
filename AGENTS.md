## 🎯 1. Философия и «Конституция» проекта
1. **Точность = Доверие.** Никаких `float64` для денежных сумм и процентов. Только `shopspring/decimal`.
2. **Doc ↔ Code Sync.** Документация и код не должны расходиться. Если меняется формула — обновляется doc-комментарий со ссылкой на источник (Копнова, ЦБ, НК РФ).
3. **Privacy by Design.** PII (Перс. данные) маскируется *до* сохранения в БД и *до* логирования. В логах только `user_hash`.
4. **Прозрачность.** Каждая финансовая функция имеет doc-comment с формулой и источником — это нужно для фичи «Показать формулу» в UI.
5. **Scope-lock.** Мы не делаем то, что не запланировано в текущем Этапе. Никаких «а давайте ещё добавим ML-категоризацию», если мы в MVP.

## ⚖️ 3. Жёсткие Архитектурные Правила
### Слои и Пакеты
- `domain/` — бизнес-сущности и инварианты. `domain.Money` (обёртка над decimal, scale=2).
- `storage/` — работа с БД. Интерфейсы репозиториев. Ошибки БД транслируются в sentinel-ошибки (`translatePgError`).
- `service/` — бизнес-логика. Оркестрация, валидация, PII-маскирование.
- `transport/http/` — хендлеры, парсинг JSON, маппинг sentinel-ошибок в HTTP-статусы.
- `mathcore/` — изолированные математические пакеты (tvm, credit, investment, tax, goals).

### Работа с Деньгами и JSON
- **ВСЕГДА** передавать деньги в JSON как **строки** (`"100.50"`), чтобы избежать float-потерь на фронте.
- Округление: `ROUND_HALF_AWAY_FROM_ZERO` (1.005 → 1.01), НЕ bankers' rounding.
- Идемпотентность: операции используют `calc_id` (формат `srv:{user_id}:{unix_nano}`). При дубле БД выкинет `23505`, сервис вернёт оригинал.

## 🧮 4. Математическое Ядро (`mathcore`) и Источники
Все расчёты ведутся в рублях (RUB). Источники истины: учебник Копнова Е.Д. «Финансовая математика», Указания ЦБ РФ, НК РФ.

| Пакет | Назначение | Критические нюансы |
|---|---|---|
| `daycount` | ACT/ACT, ACT/360 | Корректно обрабатывает year-boundary. |
| `tvm` | Simple/Compound, Fisher | `Pow` в shopspring принимает дробные степени. |
| `credit` | Аннуитет, Дифф., ПСК | **ПСК = номинальная ставка** (по ЦБ 5750-У), а НЕ эффективная! Использует BrentQ (float64 bridge). |
| `investment` | NPV, XIRR, MIRR, PI | **XIRR = эффективная годовая доходность** (в отличие от ПСК). |
| `tax` | НДФЛ, НПД, УСН, Вклады | **НДФЛ считается маржинально** (по ступеням ФЗ-257), а не «всё по верхней ставке». |
| `goals` | Sinking Fund | Аналитическое решение, без BrentQ. Учитывает инфляцию. |

## 🔒 5. Безопасность и Приватность (152-ФЗ)
- **PII Masking (`internal/pii/pii.go`):** Regex маскирует телефоны, email, паспорта, карты, ФИО.
  - ⚠️ **ГРАБЛЯ:** Go RE2 `\b` работает ТОЛЬКО с ASCII. Для кириллицы `\b` убран, ФИО ищется как «2+ ALLCAPS токенов через whitespace».
- **JWT:** Access (15 мин) и Refresh (30 дней) на **разных** секретах. Refresh single-use (atomic `UPDATE...RETURNING`). В БД хранится только `SHA-256(refresh)`.
- **Логи:** Только `applog.WithUserHash`. Email/телефоны/ФИО никогда не попадают в slog.

## 🛠 7. Критические «Грабли» (Запомнить навсегда!)
1. **`embed.FS` и Windows:** Всегда используй `path.Join` (forward-slash), а не `filepath.Join`. Иначе `fs.ReadFile` упадёт на Windows.
2. **`shopspring/decimal` v1.4.0:** 
   - `Pow(d2 Decimal) Decimal` — возвращает Decimal БЕЗ error (внутри глотает ошибки).
   - `Ln(precision int32) (Decimal, error)` — возвращает error, требует явный precision (используй 16).
3. **ПСК vs XIRR:** ПСК (кредиты) = номинальная ставка по ЦБ. XIRR (инвестиции) = эффективная доходность. Не путать!
4. **НДФЛ:** С 2025 года 5 ступеней (13% / 15% / 18% / 20% / 22%). Считается *маржинально* (slice базы), а не прогрессивно на всю сумму.
5. **Транзакции:** `SeedSystemCategories` и `SeedDefaultsForUser` вызываются внутри tx регистрации. Ошибка seed'а логируется, но НЕ откатывает регистрацию.
