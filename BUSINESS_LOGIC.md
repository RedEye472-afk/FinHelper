# 📘 Бизнес-логика «ФинПомощник»

**Формат**: А (Вход) → В (Обработка) = С (Результат)

Этот документ описывает, КАК работает каждая функция приложения. Claude Code использует эту информацию для понимания бизнес-логики, edge-cases и формул.

---

## 🔹 УРОВЕНЬ 1: Ввод, учёт, базовая аналитика

### 1. Ручной ввод операций

**Простыми словами**: Ты как будто диктуешь кассиру: «Запиши: сегодня потратил 500 на кофе». Он записывает, и твой «кошелёк в телефоне» сразу показывает, сколько осталось.

**А (Вход)**:
- Тип операции: Доход / Расход / Внутренний перевод / Конвертация валюты / Возврат
- Сумма, Валюта, Дата, Источник (какой счёт), Контрагент, Описание
- Для доходов: подтип (Зарплата/Гонорар/Подарок/Инвестиции/Возврат займа)

**В (Обработка)**:
- Валидация: сумма > 0, дата ≤ сегодня (или флаг "плановая")
- Конвертация валют по курсу ЦБ на дату операции
- Генерация уникального `calc_id` (для идемпотентности)
- **Критично**: переводы/возвраты/конвертации НЕ попадают в cashflow и налоги, только в остатки счетов
- При первом контрагенте → запрос подтверждения подтипа дохода

**С (Результат)**:
- Запись в БД с типом операции
- Обновление *чистого* cashflow и остатков по счетам
- История: "Возврат от 12.05 уменьшил расход 'Покупки', доход не создан"

**Edge-cases**:
- Отрицательная сумма → ошибка
- Дата в будущем → пометить как "плановая"
- Первый контрагент → запросить подтип дохода

---

### 2. Авто-категоризация с обучением

**Простыми словами**: Приложение как умный помощник: видит «Магнит» и говорит: «Наверное, это еда». Если ошиблось — ты поправил, и оно больше не ошибается.

**А (Вход)**:
- Контрагент, MCC-код (от банка), описание транзакции
- История правок пользователя
- Справочник правил (MCC → категория, ключевые слова → категория)

**В (Обработка)**:
- Уровень 1: MCC-коды (высокая точность, ~90%)
- Уровень 2: Ключевые слова в описании (~70%)
- Уровень 3: ML на основе правок пользователя (обучается)
- Расчёт уверенности. При <70% → запрос подтверждения
- После 3 подтверждений → создание персонального правила
- Учёт подтипов доходов (ЗП/Гонорар) для корректной аналитики

**С (Результат)**:
- Присвоенная категория + уверенность (например, "Еда: продукты, 92%")
- Если низкая уверенность → подсветка: "Проверьте категорию?"
- Лог обучения (обезличенный): пользовательский паттерн сохранён

**Edge-cases**:
- Неизвестный MCC → уровень 2
- Пользователь исправил 3 раза → создать правило
- Новый контрагент → спросить категорию

---

### 3. Сводный дашборд «Финансовое здоровье»

**Простыми словами**: Как приборная панель в машине: сразу видишь «сколько бензина», «куда едешь», «хватит ли до заправки». Только про деньги.

**А (Вход)**:
- Период (месяц/квартал/год)
- Все операции (с фильтрацией по типам)
- Цели пользователя
- Календарь будущих платежей

**В (Обработка)**:
- Баланс = Σ(реальные доходы) − Σ(реальные расходы). **Переводы исключены**
- Чистая стоимость = Σ(Активы) − Σ(Долги)
- Свободный остаток = текущий баланс − резерв под налоги/страховку/кредиты из календаря
- Прогресс целей = накоплено / цель

**С (Результат)**:
- Реальный cashflow за период
- "Свободно сейчас: 40 000₽. С учётом налогов/страховки: 18 000₽"
- Диаграмма расходов по категориям
- Прогресс-бары по целям
- Статус синхронизации счетов

**Edge-cases**:
- Нет операций за период → показать "Нет данных"
- Отрицательный баланс → красный индикатор
- Цель просрочена → жёлтый индикатор

---

### 4. Лимиты и бюджет со скользящим переносом

**Простыми словами**: Как счётчик в такси: «До конца поездки осталось 200₽». Только ты сам задаёшь лимит, и приложение следит, чтобы ты не «уехал» дальше.

**А (Вход)**:
- Бюджет по категориям (например, "Еда: 15 000₽/мес")
- Факт трат за период
- Календарь будущих платежей
- Настройка переноса (да/нет/3 мес.)

**В (Обработка)**:
- Остаток = бюджет − факт + перенос из прошлого периода (если включено)
- Прогноз перерасхода с учётом темпа трат и оставшихся дней
- Выделение "реально доступных" средств только после резерва под обязательные платежи

**С (Результат)**:
- Остатки по категориям с учётом переноса
- Алерт: "Риск перерасхода по 'Еде'. Доступно: 200₽"
- "Свободные средства после резерва: 18 000₽"

**Edge-cases**:
- Перенос отключён → остаток сгорает
- Перенос 3 мес. → накапливается до 3 месяцев
- Превышение бюджета → красный индикатор

## 🔹 УРОВЕНЬ 2: Детерминированные расчёты и планирование

### 5. Трекер целей с учётом внеплановых пополнений

**Простыми словами**: Как навигатор: «Чтобы доехать до цели к 18:00, нужно ехать 60 км/ч. Ты едешь 80 км/ч — приедешь в 17:30». Только про деньги.

**А (Вход)**:
- Цель (название, сумма, срок)
- Текущее накопление
- Регулярный взнос (ежемесячно)
- Ожидаемая доходность (если положить на вклад)
- Внеплановые поступления (бонус, подарок)
- Инфляция (опционально)

**В (Обработка)**:
- Формула аннуитета для расчёта срока/взноса (Копнова Гл. 3.3.3)
- Мгновенный пересчёт при внеплановом пополнении
- Опциональная корректировка цели на инфляцию
- Прогресс = текущее / цель

**С (Результат)**:
- Требуемый взнос/скорректированный срок
- Прогресс-бар + история: "+50 000₽ от продажи → срок −2 мес."
- Дисклеймер: "Без учёта изменения ставок и налогов"

**Edge-cases**:
- Цель уже достигнута → "Поздравляем!"
- Взнос слишком маленький → "Не успеете к сроку"
- Внеплановое пополнение → пересчёт срока

---

### 6. Калькулятор вкладов

**Простыми словами**: Как калькулятор в банке, только честнее: показывает не только «сколько будет», но и «сколько реально останется после инфляции и налогов».

**А (Вход)**:
- Сумма вклада
- Ставка (% годовых)
- Срок (месяцев/лет)
- Капитализация (ежемесячно/ежеквартально/в конце срока)
- Инфляция (из данных ЦБ/Росстата)
- Год расчёта (для версионирования правил налога)

**В (Обработка)**:
- Сложные проценты с daycount (ACT/365)
- Налог на вклад: динамический порог из справочника ФНС/ЦБ за выбранный год
- Формула Фишера для реальной доходности
- Версионирование правил (не хардкод)

**С (Результат)**:
- Итоговая сумма, проценты, налог (с указанием года и порога)
- Реальная доходность после инфляции
- Предупреждение: "Расчёт предварительный. Уточните в ЛК банка"

**Edge-cases**:
- Ставка = 0 → только сумма
- Срок = 0 → ошибка
- Доход ниже порога налога → налог = 0

---

### 7. Кредитный калькулятор

**Простыми словами**: Как честный друг: «Ты берёшь 1 млн, а вернёшь 1.13 млн. Вот расписание, вот как сэкономить, если платить чуть больше».

**А (Вход)**:
- Сумма кредита
- Ставка (% годовых)
- Срок (месяцев)
- Тип платежей: аннуитет (равные) или дифференцированный (убывающие)
- Комиссии и страховки (единые, ежемесячные)
- Параметры досрочного погашения (сумма, частота)

**В (Обработка)**:
- Аннуитет/дифф. по формулам Копновой Гл. 4.2
- ПСК включает все обязательные платежи по методике ЦБ
- Досрочка: пересчёт остатка, выбор "сокращение срока/платежа"
- Учёт daycount и комиссий в общей переплате

**С (Результат)**:
- График платежей (таблица)
- Ежемесячный платёж, ПСК, переплата
- Сценарий досрочки: "Экономия X, срок сократится на Y мес."
- Дисклеймер о влиянии скрытых условий банка

**Edge-cases**:
- Ставка = 0 → платёж = сумма / срок
- Досрочка = 0 → обычный график
- Комиссии > 0 → ПСК выше ставки

---

### 8. Анализ доступности кредита

**Простыми словами**: Как врач перед операцией: «Твоё давление в норме, но если упадёт — будет риск. Давай подберём безопасную дозу». Только про кредит.

**А (Вход)**:
- Доходы за 6–12 мес. (история)
- Обязательные расходы (еда, жильё, транспорт)
- Текущие кредиты (платежи)
- Подушка безопасности (накопления)
- Параметры нового кредита (сумма, ставка, срок)

**В (Обработка)**:
- Безопасный платёж на основе **минимального стабильного дохода**, а не среднего
- Доля платежа ≤ 30–40% от свободного остатка
- Стресс-тест: падение дохода на 20–30%
- Проверка покрытия подушкой при потере работы

**С (Результат)**:
- Диапазон безопасного платежа
- Оценка риска: "Безопасно / На грани / Критично"
- Сценарий: "При падении дохода на 30% кредит займёт 70% бюджета"

**Edge-cases**:
- Доход нестабильный → использовать минимум
- Подушка < 3 месяцев → высокий риск
- Текущие кредиты > 50% дохода → отказать

---

### 9. Ипотека vs Аренда

**Простыми словами**: Как два друга спорят: «Купить или снимать?». Приложение считает за обоих и говорит: «Если жить больше 12 лет — покупай, если меньше — снимай».

**А (Вход)**:
- Стоимость жилья
- Первый взнос (% или сумма)
- Ставка ипотеки, срок
- Страховка, налоги на имущество, ремонт
- Аренда аналогичной квартиры
- Альтернативная доходность взноса (если бы лежал на вкладе)
- Прогноз роста цен на недвижимость (опционально)

**В (Обработка)**:
- Полная стоимость ипотеки: платежи + страховка + налоги + ремонт
- Альтернативная стоимость: доход, который принёс бы первый взнос на вкладе
- Сравнение с арендой с учётом этих факторов
- Расчёт точки безубыточности

**С (Результат)**:
- Ежемесячный платёж vs Аренда
- Точка безубыточности (в годах)
- Вывод в диапазоне: "Ипотека может быть выгоднее на 5–12% при жизни >N лет"
- Список скрытых расходов с пояснениями

**Edge-cases**:
- Взнос < 20% → выше ставка
- Рост цен = 0 → аренда выгоднее
- Срок < точки безубыточности → аренда

## 🔹 УРОВЕНЬ 3: Бизнес-модуль и продвинутая аналитика

### 10. Оптимизатор расходов

**Простыми словами**: Как друг, который заглянул в твои траты и говорит: «Слушай, ты платишь за три сервиса, которыми не пользуешься. И на еду тратишь больше других. Давай посмотрим, как сэкономить».

**А (Вход)**:
- История трат за 3–6 мес.
- Подписки с датами последнего использования
- Аномалии (траты выше среднего)
- Настройки пользователя (скрывать/показывать бенчмарки)

**В (Обработка)**:
- Анализ подписок: если >60 дней нет активности → проверка на "фоновый сервис"
- Аномалии: отклонение >2σ от среднего по категории
- Бенчмарки только при явном включении пользователем

**С (Результат)**:
- Список кандидатов на отмену с контекстом: "Фоновый сервис, отмена может повлиять на данные"
- Алерты на аномалии
- Бенчмарк с дисклеймером: "Статистика не учитывает ваши обстоятельства. Ориентир, не норма"

**Edge-cases**:
- Подписка используется редко → спросить "Отменить?"
- Аномалия разовая → не алертить
- Бенчмарки отключены → не показывать

---

### 11. Оценка выгоды покупки

**Простыми словами**: Как весы: на одну чашу — «купить сейчас», на другую — «подождать». Приложение взвешивает все «за» и «против» и показывает, что тяжелее.

**А (Вход)**:
- Стоимость покупки
- Свободный остаток (сколько есть сейчас)
- Параметры кредита/накопления
- Инфляция товара (риск изменения цены)

**В (Обработка)**:
- 3 сценария: сейчас (влияние на бюджет), копить (доход на вкладе), кредит (переплата + нагрузка)
- Альтернативная стоимость денег сейчас
- Корректировка на инфляцию/дефляцию товара

**С (Результат)**:
- Сравнительная таблица сценариев
- Вывод: оптимальный путь с обоснованием
- Учёт рисков: "Если цена вырастет на 10%, копить станет менее выгодно"

**Edge-cases**:
- Свободный остаток < стоимости → только кредит/копить
- Инфляция высокая → купить сейчас
- Ставка кредита высокая → копить

## 🔹 УРОВЕНЬ 4: Прогнозы, сценарии, ИИ-слой

### 12. Симулятор «Что если»

**Простыми словами**: Как игра «Судьба»: «А что, если бы ты получил премию?». Приложение показывает, как изменится твоя финансовая жизнь при разных сценариях.

**А (Вход)**:
- Текущие параметры (доходы, расходы, цели, кредиты)
- Изменяемый фактор (доход ±X%, инфляция, курс)
- Горизонт прогноза (3/6/12 мес.)

**В (Обработка)**:
- Пересчёт всех зависимых переменных на основе детерминированных формул ядра
- Визуализация разницы "было/стало"
- Расчёт влияния на цели и cashflow **без генерации новых чисел**

**С (Результат)**:
- Графики и цифры изменений
- Кнопка "Применить к плану"
- Дисклеймер: "Модель не гарантирует реальное изменение рынка/доходов"

**Edge-cases**:
- Изменение = 0 → нет изменений
- Горизонт > 12 мес. → предупреждение о точности
- Применить к плану → обновить цели

---

### 13. Объясняющий ИИ (строгие guardrails)

**Простыми словами**: Как учитель, который не просто даёт ответ, а объясняет: «Вот почему так получилось, вот формула, вот аналогия из жизни». Но всегда напоминает: «Проверь сам, я не гарантирую».

**А (Вход)**:
- Результаты расчётов ядра (`result_json`)
- Контекст правил (`context_rules`)
- Профиль пользователя (`user_profile`)

**В (Обработка)**:
- LLM получает ТОЛЬКО числа и правила. **Запрет на генерацию расчётов/советов "купить/продать"**
- Паттерн ответа: Наблюдение → Возможная причина → Вопрос → Дисклеймер
- Обязательная кнопка "Показать формулу"

**С (Результат)**:
- Текстовое пояснение простым языком
- Кнопка раскрытия математической основы
- Футер: " Примерная аналитика. Проверьте расчёты. Не является инвестиционной/налоговой рекомендацией."

**Edge-cases**:
- LLM сгенерировала число → reject + fallback
- Нет дисклеймера → reject + fallback
- Совет "купить/продать" → reject + fallback

## 🔹 УРОВЕНЬ 5: Комплаенс, экспорт, масштабирование

### 14. Экспорт и отчётность

**Простыми словами**: Как копировать страницу из тетради: выбираешь, что копировать, в каком виде, и получаешь файл, который можно отправить бухгалтеру или в налоговую.

**А (Вход)**:
- Период (месяц/квартал/год)
- Тип отчёта: «Для себя», «Для бухгалтера», «Для ФНС», «Для банка»
- Формат: PDF / CSV / Excel / JSON
- Фильтры: только определённые категории/счета/проекты

**В (Обработка)**:
- Сбор данных за период из базы
- Форматирование под выбранный шаблон (1С, ФНС, простой список)
- Маскирование чувствительных данных в логах и выгрузках ([PERSON], [MEDICAL], [LEGAL])
- Добавление шапки-дисклеймера во все налоговые формы

**С (Результат)**:
- Файл для скачивания
- Заголовок в налоговых черновиках: "Предзаполненная форма для проверки. Проверьте данные перед подачей."
- Подтверждение экспорта с хэшем файла

**Edge-cases**:
- Большой период → предупреждение о размере
- Налоговая форма → дисклеймер обязателен
- PII в данных → маскирование

### 15. Панель согласий и аудит 152-ФЗ

**Простыми словами**: Как ключи от квартиры: ты видишь, кому дал ключ, можешь забрать его обратно, и есть журнал, кто и когда заходил. Только про доступ к твоим финансовым данным.

**А (Вход)**:
- Запрос (просмотр/отзыв/удаление)
- `user_hash` (анонимный идентификатор)
- Тип данных/доступа

**В (Обработка)**:
- Проверка прав: только владелец аккаунта может менять согласия
- Для отзыва: немедленная инвалидация токена доступа к источнику
- Для удаления: мягкое удаление (30 дней на восстановление) → полное удаление
- Логирование в immutable ledger (обезличенно, хранение 3 года)

**С (Результат)**:
- Список активных доступов с кнопками отзыва
- Подтверждение отзыва: «Доступ к Тинькофф отозван. Данные перестанут обновляться.»
- Подтверждение удаления: «Запрос на удаление принят. Данные будут удалены через 30 дней.»
- Аудит-трек: "28.05.2026 14:32 — отзыв доступа к Тинькофф (hash: ...)"

**Edge-cases**:
- Отзыв доступа → немедленная инвалидация токена
- Удаление → soft delete 30 дней → hard delete
- Аудит-лог → только INSERT, без UPDATE/DELETE
