# OSS Integrator

Этот скилл позволяет AI-агенту профессионально подходить к интеграции открытого кода в проект, экономя время и избегая типовых ошибок.

## Когда использовать

- Проекту нужен функционал, который уже реализован в известных open-source библиотеках
- Обнаружен повторяющийся код, который можно заменить готовым решением
- Нужно добавить интеграцию с популярными сервисами/API
- Требуется валидация, парсинг, шифрование, работа с форматами и т.д.
- Нужна утилита/хелпер, которая есть в thousands репозиториев

## Когда НЕ использовать

- Функционал тривиален (написать самому быстрее, чем разбираться в чужом коде)
- Зависимость принесёт больше кода, чем решает проблем (overengineering)
- Лицензия несовместима с лицензией проекта
- Репозиторий заброшен, не поддерживается, имеет known vulnerabilities
- Код не имеет тестов или документации

## Процесс работы агента

### Шаг 1: Анализ потребности

Прежде чем искать — пойми задачу:
1. Определи конкретную проблему/функцию, которую нужно решить
2. Можно ли решить её стандартной библиотекой языка?
3. Есть ли уже в проекте похожая реализация? (grep по коду)
4. Оцени сложность самостоятельной реализации (часы/дни)

### Шаг 2: Поиск кандидатов

Используй websearch с конкретными запросами:
- `"github <technology> <feature> library"`
- `"best open-source <language> <feature> 2025 2026"`
- `"awesome <feature>"` (списки awesome-*)
- Поиск по GitHub: `site:github.com <language> <feature>`

Проверь `@public-datasets` — возможно, нужные данные уже есть.

Установи приоритеты:
- Официальные SDK и библиотеки от известных компаний
- Репозитории с >1000 stars и активным мейнтейнером
- Решения, встроенные в экосистему (npm, PyPI, cargo, etc.)
- Решения с наиболее свежими коммитами (< 6 месяцев)

### Шаг 3: Оценка безопасности и качества

Для каждого кандидата выполни проверку:

**3.1 Безопасность (обязательно):**
- Просмотри `package.json` / `Cargo.toml` / `requirements.txt` / `go.mod` — какие зависимости тянет библиотека
- Есть ли известные CVE? (`npm audit`, `cargo audit`, `safety check`)
- Не требует ли код опасных permissions (доступ к ФС, сеть, exec)?
- Есть ли обработка пользовательского ввода? Не допускает ли injections?
- Анализируй файлы с чувствительными операциями (eval, exec, File operations, network)

**3.2 Качество кода:**
- Есть ли тесты? (`__tests__/`, `test/`, `*.test.ts`, `*.spec.ts`)
- Какое покрытие? (запусти `npx jest --coverage`, если jest)
- Есть ли CI/CD? (проверь `.github/workflows/`, `.travis.yml`)
- Архитектура: читаемый код? разделение ответственности? типизация (TypeScript)?
- Код-стайл: соответствует ли стандартам сообщества?

**3.3 Активность и поддержка:**
- Последний коммит
- Количество contributors
- Open/closed issues ratio
- Есть ли response от мейнтейнеров в issues
- Есть ли релизы и changelog

**3.4 Лицензия:**
- MIT, Apache 2.0, BSD — безопасны для коммерции
- GPL, AGPL — могут быть ограничения
- Проверь совместимость с лицензией проекта

### Шаг 4: Принятие решения

Оценка по системе +1/0/-1:

| Критерий | +1 | 0 | -1 |
|---|---|---|---|
| Лицензия | MIT/Apache/BSD | LGPL/MPL | GPL/не указана |
| Stars | >1000 | 100-1000 | <100 |
| Последний коммит | < 3 мес | 3-12 мес | > 12 мес |
| Тесты | есть с покрытием | есть, без покрытия | нет |
| Зависимости | минимальны | умеренны | тяжеловесны |
| CVE | нет | minor | critical |

**Решение:**
- **+4 и выше** — можно интегрировать
- **0 до +3** — нужно дополнительное обоснование
- **< 0** — лучше написать самому или искать альтернативу

Если решение неочевидно — **спроси пользователя**.

### Шаг 5: Интеграция

**5.1 Установка:**
```
npm install <package>
# или
cargo add <crate>
# или
pip install <package>
```

**5.2 Адаптация под проект:**
- Скопируй только нужную часть (если лицензия позволяет) — `extract` подход
- Оберни в абстракцию проекта (adapter/facade pattern), чтобы можно было легко заменить
- Не используй библиотеку напрямую по всему коду — создай wrapper
- Удали дублирующийся код проекта, который заменила библиотека

**5.3 Конфигурация:**
- Добавь `.env` переменные если нужно
- Проверь настройки по умолчанию — безопасны ли они для production
- Настрой logging/debug режим библиотеки

### Шаг 6: Верификация

После интеграции:
1. Запусти существующие тесты проекта — всё ли проходит?
2. Проверь, что старая функциональность не сломалась
3. Напиши минимальный тест на интеграцию (smoke test)
4. Проверь импорты — нет ли dead imports
5. Запусти линтер и форматтер

### Шаг 7: Документирование

Обнови:
- В файл зависимостей (комментарий зачем добавлена библиотека)
- В README или CHANGELOG — что изменилось
- Если заменил старый код — отметь deprecated и план миграции

## Структура оценки (шаблон для отчёта)

```
## Оценка: <repo-name>

**URL:** https://github.com/...
**Лицензия:** MIT
**Звёзды:** 5200
**Последний коммит:** 2026-03-15
**Тесты:** ✅ (coverage ~85%)
**CVE:** ❌ нет известных
**Зависимости:** 3 прямые, 12 транзитивных (адекватно)

**Вердикт:** ✅ Интегрировать | ⚠️ Нужно обсуждение | ❌ Не использовать

**Обоснование:**
...
```

## Полезные команды для оценки

```powershell
# Клонировать и посмотреть размер
git clone --depth 1 <url> <temp_dir>

# Поиск опасных паттернов в репо
rg -n "(eval|exec|unescape|dangerouslySetInnerHTML|innerHTML\s*=)" --include "*.{js,ts,jsx,tsx}"

# Размер node_modules если npm пакет
npm ls --depth=0

# Проверка лицензии
rg -i "license" --include "*.md" --include "LICENSE*" -g "!node_modules"

# Посмотреть структуру
Get-ChildItem -Recurse -Depth 2 | Select-Object FullName

# Проверить тесты
Test-Path "test/", "__tests__/", "*.test.ts", "*.spec.ts"
```
